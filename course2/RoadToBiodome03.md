#### 시간 복잡도의 정의와 필요성

- 정의: 입력 크기(N)가 커질 때 알고리즘의 실행 시간이 얼마나 증가하는지를 나타내는 척도

- 필요성
	- 1. 입력이 커졌을 때 프로그램이 버틸 수 있는지 미리 예측 가능하다.
	- 2. 서로 다른 알고리즘/구현 방식을 객관적으로 비교 가능하다.
	- 3. 하드웨어에 의존하지 않는 일관된 성능 기준을 제공한다.
	- 4. 병목 지점을 미리 찾아내는 근거가 된다.

#### Big O 표기법의 개념과 상황별 예시

- Big O 표기법 개념: 알고리즘의 시간 복잡도를 표현하는 표기법. 입력 크기(N)이 커질 때, 연산 횟수가 늘어나는 상한선을 표현

- 핵심 규칙
	- 상수는 무시한다
		- 연산이 2N, 100N 번 되더라도 N 이 충분히 커지면 결국 N에 비례해서 늘어난다는 본질이 똑같기 때문에 둘 다 O(N)으로 쓴다.
		- 계수는 표기법에서 제거
	- 가장 영향력이 큰 항(최고차 항)만 남긴다.
		- 연산 횟수가 n² + n + 100 이라면, n의 값이 커지면 커질수록 n²이 나머지 항들의 값을 무시해도 될 만큼 커지기 때문에 O(n²)으로 표현한다.
	- 최악의 경우(worst case)를 기준으로 삼는다.

- 상황별 예시
	- 1. O(1) - 상수 시간
		- 입력의 크기와 무관하게 항상 일정한 횟수만 연산
		- 배열이나 리스트에서 인덱스로 직접 접근하는 연산이 O(1)예시
	
	- 2. O(n) - 선형 시간
		- 입력 크기(N)에 정비례해서 연산 횟수가 늘어난다.
		- 대표적인 예시로 반복문(for, while)이 존재

	- 3. O(n²) - 이차 시간
		- 입력이 늘어날 때 연산 횟수가 n²에 비례
		- 반복문 안에 반복문(중첩 루프)구조 혹은 매 반복마다 반복문 안에 O(n) 작업이 있는 경우

	- 4. O(log n) — 로그 시간
		- 단계마다 처리할 대상이 절반씩 줄어드는 경우
		- 대표적인 예시로 이진 탐색(Binary Search)이 존재
	
	- 5. O(n log n) - 선형 로그 시간
		- 입력 크기(N)에 비례해서 연산 횟수가 로그 배만큼 늘어난다.
		- 대표적인 예시로 퀵 정렬(Quick Sort), 합병 정렬(Merge Sort), 힙 정렬(Heap Sort)이 존재
	
	- 6. O(2ⁿ) - 지수 시간
		- 입력 크기(N)가 늘어날 때 연산 횟수가 2ⁿ에 비례
		- 대표적인 예시는 피보나치(Fibonacci) 수열이 존재

#### 문제01 코드의 시간 복잡도 분석

- 시간복잡도 : O(n)

- 입력 개수(N)일 때, 아래 코드가 N번 돌 때 시간 복잡도 O(n) + O(1) = O(n)

```
for (int i = 0 ; i < args.length; i++){ // N 번 반복
    int num = Integer.parseInt(args[i]);
    if (num < 0 || num > 1000){ ... }
    list.add(num);
    arr[num] += 1;
}


public static int findNumber(int[] arr){
    for (int i = 0; i < 1001; i++){ // 최악의 경우 1001번이 최대 -> 입력 개수(n)와 무관한 값
        if (arr[i] == 1){
            return i;
        }
    }
    return -1;
}
```

#### 병목 현상 분석 및 최적화 방안

- 문제01 병목 현상
	- 문제01은 이미 O(n)의 시간 복잡도를 가지고 있어 구조상의 병목은 없는 상태입니다. 다만 ArrayList<Integer>에 값을 저장할 때 마다 int -> Integer 오토박싱이 발생해서 미세한 오버헤드는 있지만 사실상 무시해도 상관없을 수준이라 생각합니다.

- 문제02 병목 현상
	- 검사 순서 문제 : Java에서 논리 연산, 즉 아래코드에서 || 연산 시 왼쪽 부터 short-circuit 이 발생하기 때문에 만약 입력으로 100만자가 넘어가는 값이 들어오면 정규표현식(O(n)) 검증 부분이 먼저 동작하고 길이체크(O(1))를 뒤에 하는 병목이 발생 가능하다.
	```
	if (!str.matches("[a-zA-Z0-9가-힣ㄱ-ㅣ\\s]+") || (str.length() < 2 || str.length() > 1000000))
	```

- 문제02 최적화 방법
	- 길이체크 조건을 먼저 검증하고 정규식 체크 조건을 뒤로 보내면 해결 가능하다
	```
	if ((str.length() < 2 || str.length() > 1000000) || !str.matches("[a-zA-Z0-9가-힣ㄱ-ㅣ\\s]+"))
	```
	- 다만 이렇게 처리를 하더라도 전체적인 시간 복잡도가 변하지는 않는다.
